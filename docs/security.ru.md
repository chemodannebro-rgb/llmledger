# Модель безопасности

Не имеющее зависимостей ядро `llm-burnwatch` читает и пишет локальные
файлы и печатает в stdout/stderr. Ни одна из команд `report`/`demo-data`/
`schema`/`status`/`validate`/`dashboard`/`detect`/`train`/`budget` не делает
сетевых вызовов — это гарантирует
`test_core_commands_make_no_network_attempts` (`tests/test_cli.py`),
который патчит `socket.socket`, чтобы он бросал исключение при любой
попытке открыть сокет.

Эта страница — читаемая версия
[`SECURITY.md`](https://github.com/chemodannebro-rgb/llm-burnwatch/blob/main/SECURITY.md)
и раздела `ARCHITECTURE.md` «Network boundaries» — вся модель доверия в
одном месте, без новых гарантий.

## Сетевые/процессные границы вкратце

| Команда | Граница | Отключено по умолчанию? |
|---|---|---|
| `pricing import <url>` | Скачивает JSON-файл с ценами по `http(s)://` | Да — только при явном URL-аргументе |
| `detect --follow --webhook-url`/`--slack-webhook-url` | Отправляет каждый новый сработавший алерт как JSON (или Slack-совместимый payload) на указанный вами URL | Да — только с флагом/переменной окружения, и только в режиме `--follow` |
| `detect --follow --telegram-bot-token`+`--telegram-chat-id` | Отправляет каждый алерт в Telegram Bot API (`api.telegram.org`, хост зафиксирован приёмником) | Да — те же условия, что у webhook/Slack |
| `detect --follow --exec-sink <command...>` | Запускает **локальную команду** (не сетевой вызов) на каждый алерт | Да — только с флагом |

Ничто из перечисленного никогда не запускается неявно, а разовый `detect`
вообще не касается ни одного из этих механизмов. `detect --follow` сам по
себе исключён из теста «без сети» (это долгоживущий poll-цикл);
эквивалентная гарантия для него — что ни один сокет не открывается и ни
один процесс не порождается, если не заданы флаги приёмников, —
проверяется отдельно, патчингом и `socket.socket`, и `subprocess.Popen`.

## Граница доверия `pricing import <url>`

`pricing import <source>` — единственное явное, добровольное исключение
из гарантии «без сетевых вызовов». Оно никогда не запускается неявно —
только когда вы вызываете именно эту подкоманду с URL.

**Что делает:** скачивает `<source>` по `http(s)://` (таймаут 10 секунд,
лимит ответа 10 МБ, отклоняет любую другую схему URL вроде `file://`, и
отказывается следовать редиректу, понижающему `https://`-источник до
обычного `http://`), строго парсит как JSON (отклоняя `Infinity`/`NaN`/
не-объектные payload'ы), извлекает только числовые поля цены за токен и
записывает результат в `~/.config/llm-burnwatch/pricing.json`. Скачанное
содержимое никогда не исполняется — оно читается как данные, точно так
же, как `report`/`dashboard`/`detect` уже читают встроенный `pricing.json`.

**Чему вы доверяете:** содержимому по указанному вами URL. Вредоносный
или скомпрометированный URL может подсунуть завышенные или заниженные
тарифы по моделям, незаметно искажая все последующие расчёты стоимости,
пока вы не импортируете корректный файл заново. Это риск данных (точности
цен), а не риск выполнения кода.

## Граница доверия файла состояния `detect --follow`

`--follow` сохраняет прогресс (уже обработанное байтовое смещение плюс
текущее скользящее окно) в
`<log-file>.llm-burnwatch-follow-state.json`, записываемый атомарно
(`tempfile.mkstemp` + `os.replace`) рядом с логом — процесс, убитый
посреди записи, никогда не оставляет наполовину записанный файл
состояния.

При загрузке верхнеуровневая форма файла проверяется до того, как его
содержимому начинают доверять; отсутствующий, нечитаемый, не валидный
JSON или файл неправильной формы никогда не фатален — `--follow`
предупреждает и начинает заново с начала лога вместо падения.

**От чего это не защищает:** файл состояния не несёт проверки
целостности, аналогичной sha256 у реестра моделей ниже. Тот, у кого есть
доступ на запись к нему, может вручную отредактировать байтовое смещение
(заставив `--follow` пропустить или повторно прочитать части лога) либо
внедрить произвольный JSON в сохранённое окно. Это тот же уровень
доверия, что и у самого лог-файла — если вы и так не доверяете всем, у
кого есть доступ на запись к директории `--log-file`, файл состояния
follow не даёт более сильной гарантии.

## Граница доверия приёмников алертов `detect --follow`

Приёмники алертов (`WebhookSink`/`SlackSink`/`TelegramSink`/`ExecSink`)
отправляют каждый новый сработавший алерт в пункт назначения, который
настраиваете *вы*. Ни один не запускается неявно, ни один не работает при
разовом `detect`, а сбой в одном приёмнике перехватывается, сообщается
через `warn()` и никогда не останавливает ни poll-цикл, ни остальные
настроенные приёмники.

**Webhook/Slack**: HTTP(S) POST с той же дисциплиной `urllib.request` +
10-секундного таймаута, что и у `pricing import`, включая отклонение
любой схемы, отличной от `http(s)://`, до попытки соединения.
Предпочитайте переменные окружения
`LLM_BURNWATCH_WEBHOOK_URL`/`LLM_BURNWATCH_SLACK_WEBHOOK_URL` вместо
CLI-флагов для URL, содержащего секретный токен, поскольку аргументы
командной строки видны другим локальным пользователям через `ps`.

**Telegram** внутренне использует `WebhookSink` — фиксированный endpoint
`https://api.telegram.org/bot<token>/sendMessage` (хост зашит в код, а не
передаётся вызывающей стороной). Предпочитайте
`LLM_BURNWATCH_TELEGRAM_BOT_TOKEN`/`LLM_BURNWATCH_TELEGRAM_CHAT_ID` вместо
CLI-флагов по той же причине видимости через `ps`.

**Exec sink** — самый рискованный из четырёх: он запускает *указанную вами
локальную команду*, передавая JSON алерта в её **stdin** (никогда не в
argv, поскольку argv видим через `ps`/`/proc/<pid>/cmdline`). `shell=False`
зашит в `ExecSink.send()` — это не параметр, который вы или будущий
вызывающий код можете переопределить, — поэтому JSON алерта никогда не
конкатенируется в строку, которую заново парсит shell.

**От чего это не защищает:** `shell=False` гарантирует только то, что этот
процесс не передаёт вашу командную строку shell'у. Если сама
настраиваемая вами команда интерпретирует свой stdin как код/шаблоны
(например, `sh`, читающий скрипт из stdin, скрипт с собственным
`eval`/шаблонизацией), эта команда всё ещё может исполнить содержимое,
производное от алерта. Указывайте в `--exec-sink` только команду,
трактующую свой stdin как непрозрачную строку.

## Граница доверия реестра моделей

`llm-burnwatch train` (extra `[anomaly]`) пишет версионированный реестр
моделей в `models/vN/`: файл `model.skops` плюс `metadata.json`,
записывающий sha256-хэш `model.skops`. `detect` читает этот реестр
обратно для своей ML-перекрёстной проверки.

**От чего это защищает:**

- **Повреждение или случайная подмена** — `load_model()` пересчитывает
  sha256 `model.skops` и отказывается загружать файл, если он не
  совпадает с `metadata.json`.
- **Произвольное выполнение кода через десериализацию** — модели
  сериализуются через `skops.io`, а не `pickle`. В отличие от `pickle`,
  `skops` по построению отказывается собирать любой тип вне явного
  доверенного списка, поэтому подделанный файл отклоняется при загрузке
  (`load_model()` также проверяет `skops.io.get_untrusted_types()` перед
  десериализацией).

**От чего это не защищает:** от скоординированной подмены тем же
автором/коммитом. Тот, кто контролирует репозиторий (или CI-задачу,
запускающую `train`), может заменить `model.skops` другой моделью и
просто пересчитать sha256, чтобы он совпал. Проверка целостности
обнаруживает только несовпадение между двумя файлами — она не может
отличить легитимный запуск `train` от вредоносного той же стороной. Это
ограничение корня доверия, а не баг: кому вы доверяете трогать реестр —
вопрос процесса (например, ревью диффа), а не то, что может решить
контрольная сумма. `load_model()` печатает предупреждение об этом при
каждой загрузке модели.

## Сообщить об уязвимости

`llm-burnwatch` — проект на раннем этапе — нет SLA, но сообщения всё
равно приветствуются. Откройте issue на GitHub с описанием проблемы; для
этого проекта нет выделенного контакта по безопасности или канала
приватного раскрытия, так что считайте любой открытый issue публичным с
самого начала.
