# Детектор правил

**Ловит:** явное нарушение политики, настроенной *вами* — модель, которая
не должна использоваться, отдельный вызов, обошедшийся дороже, чем
следовало, целую трассу (многошаговый прогон агента), обошедшуюся дороже,
чем следовало. В отличие от всех остальных детекторов, это не
статистическое суждение; это «нарушен лимит, который вы задали», поэтому
каждый алерт от него — `severity="critical"`.

**Всегда доступен, без зависимостей. Включён по умолчанию** — но остаётся
полностью бесшумным, пока вы не настроите хотя бы одно правило. Нет
безопасного универсального значения по умолчанию для «какие модели
разрешены» или «сколько должен стоить вызов», поэтому ненастроенный
`RulesDetector` — намеренный no-op, а не детектор со встроенными
значениями по умолчанию.

## Три проверки

Настраиваются через CLI-флаги команды `detect`:

| Флаг | Вид алерта | Что проверяет |
|---|---|---|
| `--allowed-models <model> [<model> ...]` | `model_not_allowed` | `model` каждой записи входит в список разрешённых |
| `--max-call-cost <usd>` | `call_cost_exceeded` | `cost_micros` ни одного отдельного вызова не превышает это значение |
| `--max-trace-cost <usd>` | `trace_cost_exceeded` | Сумма `cost_micros` по всем записям с одним `trace_id` не превышает это значение |

Для `--max-trace-cost` алерт указывает на конкретную запись, чья стоимость
перевела накопленную сумму трассы за лимит — необязательно на самый
дорогой отдельный вызов в трассе, а именно на момент, когда лимит был
фактически пересечён, чтобы было видно, какой шаг его нарушил.

## Пример

```bash
llm-burnwatch detect --log-file calls.jsonl \
  --allowed-models gpt-4o-mini claude-3-5-haiku \
  --max-call-cost 0.50 \
  --max-trace-cost 5.00
```

## Обнаружение vs принуждение

Этот детектор постфактум: он помечает нарушение уже после того, как оно
произошло, при запуске `report`/`detect` по логу. Если нужно остановить
зациклившийся агент *до* того, как он пересечёт лимит стоимости, см.
[`CostTracker.guard()`](../budget-vs-guard.ru.md) — эти два инструмента
решают разные задачи и свободно сочетаются.

## Известные ограничения

- `--max-trace-cost` видит только записи с явным `trace_id` — вызовы,
  залогированные без него, невидимы для этой проверки (но по-прежнему
  проверяются `--allowed-models`/`--max-call-cost`, которым `trace_id` не
  нужен).
- Нет поддержки файла конфигурации — правила живут в командной строке
  `detect` (или там, где ваш собственный скрипт напрямую создаёт
  `RulesDetector`), а не в сохраняемом файле политики, как у `budget set`.
