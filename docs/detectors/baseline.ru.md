# Baseline-детектор

**Ловит:** один вызов, чьи счётчики токенов или стоимость необычны для его
пары `(label, model)` — смену шаблона промпта, внезапно утроившую длину
ответа, смену модели, незаметно изменившую тарификацию, разовый
«убежавший» ответ.

**Всегда доступен, без зависимостей. Включён по умолчанию.**

## Математика

Для каждого из `input_tokens`, `output_tokens`, `cost_micros` и
`cached_input_tokens` детектор вычисляет модифицированный z-score
Иглевича и Хоглина (Iglewicz & Hoaglin, 1993):

```
M_i = 0.6745 * (x_i - median) / MAD
```

относительно истории вызовов с той же парой `(label, model)` — median/MAD
(median absolute deviation, медианное абсолютное отклонение), а не
mean/stdev, чтобы выбросы, уже присутствующие в истории, не раздували
разброс и не маскировали новые, как это произошло бы с z-score на
mean/stdev.

Вызов помечается **аномальным** по признаку, когда `|M_i|` превышает порог
**3.5** — стандартное отсечение из той же статьи, не подобранное на глаз
под демо-данные этого проекта.

## Плавная деградация при нехватке истории

- Если у точной пары `(label, model)` меньше **5** записей
  (`MIN_GROUP_SAMPLES`), детектор откатывается на статистику модели по всем
  label.
- Если и этого недостаточно, вызов помечается как `insufficient_data`
  (severity `info`), а не молча считается нормальным или отбрасывается. Это
  отдельный вид алерта именно для того, чтобы отличать «ничего необычного»
  от «данных недостаточно, чтобы судить».
- Признак, у которого эталонный MAD равен ровно `0` (все предыдущие вызовы
  имели одинаковое значение), не может дать обычный z-score; вместо этого
  детектор помечает любое значение, отличающееся от этой (нулевого разброса)
  медианы, как «экстремальное отклонение» — замена модифицированного
  z-score для случая, который иначе дал бы бесконечный score.

## Настройка

Порог намеренно не вынесен в CLI-флаг — это параметр статистической
корректности (стандартное значение из литературы по робастной статистике),
а не вопрос вкуса. Если для вашего трафика нужен другой порог, создавайте
`BaselineDetector(threshold=...)` напрямую, а не через CLI (см.
`detectors/baseline_detector.py`).

## Известные ограничения

- Работает только на уровне одного вызова — *устойчивый* рост, который на
  каждом отдельном вызове остаётся ниже порога, но накапливается за много
  вызовов, — не задача этого детектора; см. [CUSUM-детектор](cusum.ru.md).
- Не учитывает ожидаемую *частоту* вызовов — всплеск иначе нормальных по
  размеру вызовов вообще не сработает на этом детекторе; см.
  [Детектор частоты](frequency.ru.md).
- Диагностическая подсказка, а не доказательство — помеченный вызов
  статистически необычен, но не подтверждён как ошибка, а реальная
  проблема всё ещё может проскользнуть под порогом. `report`/`detect`
  печатают эту оговорку, а также дату `last_updated` данных о ценах, при
  каждом запуске.

## Опционально: перекрёстная проверка ML

При установленном extra `[anomaly]` и обученной модели
(`llm-burnwatch train`) `detect` дополнительно прогоняет `IsolationForest`
по тем же относительным-для-группы признакам как второе мнение. `detect`
также сравнивает текущую статистику по группам с записанной на момент
обучения и предупреждает, если они разошлись (более чем на **2×**
эталонного MAD, `DRIFT_MULTIPLIER`) — сигнал, что `llm-burnwatch train`
пора перезапустить. Оба слоя — диагностические подсказки, а не гарантия.
