# Детектор частоты

**Ловит:** зациклившегося агента, или любой другой всплеск вызовов, который
аномально частый — единственный детектор, смотрящий на *сколько* вызовов
произошло за временное окно, а не на то, во что обошёлся каждый вызов.

**Всегда доступен, без зависимостей. Отключён по умолчанию** — `detect`
сам решает, включать ли его для конкретного лога, в зависимости от того,
достаточно ли в логе календарного охвата для сезонного сравнения.

## Почему отключён по умолчанию

Без понятия об ожидаемом объёме вызовов по времени суток/дню недели
рутинный всплеск «каждое утро понедельника» статистически неотличим от
зациклившегося агента, вышедшего из-под контроля. Вместо того чтобы либо
смириться с такой долей ложных срабатываний, либо вообще не поставлять
этот детектор, он отключён по умолчанию, и `detect` включает его только
когда в логе достаточно истории, чтобы сезонная база была осмысленной.

## Математика

Вызовы группируются в фиксированные окна по **60 секунд**
(`FREQUENCY_WINDOW_SECONDS`) в рамках каждой группы `(label, model)`, плюс
одна «глобальная» группировка на весь лог, которая ловит распределённый
по многим разным label/model всплеск, который история ни одной отдельной
группы не заметила бы.

Для каждого окна детектор вычисляет тот же модифицированный z-score, что
и [baseline-детектор](baseline.ru.md) (`0.6745 * (count - median) / MAD`)
по истории количества вызовов в окнах этой группы, и помечает всплеск при
`z > 3.5` (`FREQUENCY_Z_THRESHOLD`).

**Абсолютный предохранитель** помечает окно независимо от z-score, как
только число вызовов в нём достигает **100**
(`FREQUENCY_ABS_CALLS_PER_WINDOW`) — это покрывает два случая, которые
z-score не может: самый первый всплеск, для которого вообще нет
предыдущих окон для сравнения, и всплеск настолько выбивающийся из
истории, что единственным, что могло бы его поймать, иначе стал бы откат
«экстремального отклонения» при нулевом MAD.

## Сезонные (день недели × час суток) базы

Как только лог охватывает минимум **14 календарных дней**
(`MIN_SEASONAL_SPAN_DAYS`, проверяется по *диапазону* дат, а не количеству
записей), каждое окно дополнительно сравнивается с историей своего же
бакета `(день недели, час)` — построенной только из *других календарных
дат*, разделяющих этот день недели/час, никогда из собственной даты окна.
Именно это позволяет повторяющемуся всплеску «каждый понедельник в 9
утра» перестать помечаться, как только он случался достаточно часто,
чтобы стать ожидаемым паттерном для этого слота, при этом всё ещё ловя
всплеск, который пока нов — или необычно велик даже для обычно загруженного
утра понедельника. Бакет `(день недели, час)`, у которого ещё нет истории
на 5 образцов (`MIN_GROUP_SAMPLES`) из других дат, откатывается на плоское
сравнение по всей группе.

## Настройка

`z_threshold`/`abs_threshold` не являются CLI-флагами — создавайте
`FrequencyDetector(z_threshold=..., abs_threshold=...)` напрямую, если
вашему трафику нужна другая чувствительность (см.
`detectors/frequency_detector.py`).

## Известные ограничения

- Пакетный/офлайн, как и baseline-детектор — окна строятся из всей входной
  последовательности сразу, а не накапливаются постепенно. В
  `detect --follow` это означает, что всё скользящее окно перебакетируется
  заново на каждом poll (см. [Производительность](../performance.ru.md) о
  том, во что это обходится на практике).
- Записи с отсутствующей или нераспознаваемой временной меткой нельзя
  поместить в окно, и они молча исключаются.
- Ниже порога 14-дневного сезонного охвата этот детектор фактически вообще
  не работает (либо работает только с плоским, несезонным сравнением, в
  зависимости от того, как его вызывает `detect`) — зациклившийся агент в
  совсем новом логе не будет пойман одной только частотой; тем временем
  по-прежнему активны детекторы [baseline](baseline.ru.md)/
  [rules](rules.ru.md).
