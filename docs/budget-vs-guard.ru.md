# `budget` vs `guard()`

У llm-burnwatch есть два способа думать о «не трачу ли я слишком много»,
решающих разные задачи. Ни один не заменяет другой, и они свободно
сочетаются.

| | `budget`/`BudgetDetector` | `CostTracker.guard()` |
|---|---|---|
| **Компромисс** | Обнаружение | Принуждение |
| **Область** | Межпроцессная — читает весь лог | Внутрипроцессная — только вызовы текущего Python-процесса |
| **Временной масштаб** | Месячный (календарный месяц UTC) | На блок `with` (одна трасса) |
| **Память** | Сохраняется (`budget.json`, на диске) | Только в памяти, забывается при выходе из блока |
| **Что происходит при нарушении** | Алерт (`budget_pace_warning`/`budget_exceeded`); ничего не останавливается | `BudgetExceededError` бросается из вызова `log_call()`/адаптера, пересёкшего лимит |
| **Настраивается через** | `llm-burnwatch budget set --monthly <usd> --warn-at <fraction>` | `tracker.guard(max_usd_per_trace=..., max_calls_per_trace=...)` |

## `budget` — постфактум-обнаружение между процессами

```bash
llm-burnwatch budget set --monthly 100 --warn-at 0.8
```

После настройки `report` печатает трату с начала месяца и линейный
прогноз темпа, а `detect` получает вид алерта `budget`. Это сообщает вам,
что *месяц* тяготеет к превышению бюджета во всех процессах, пишущих в
лог — но только постфактум, с той периодичностью, с которой вы запускаете
`report`/`detect` (или `detect --follow`). Ничто здесь не останавливает
вызов от того, чтобы произойти. Математику прогноза см. на странице
[Детектор бюджета](detectors/budget.ru.md).

## `guard()` — принуждение в реальном времени, внутри процесса

```python
tracker = CostTracker("calls.jsonl")

with tracker.guard(max_usd_per_trace=1.0, max_calls_per_trace=20) as trace_id:
    for step in agent_loop():
        response = openai_client.chat.completions.create(...)
        tracker.log_openai_response(response, label="agent-step", trace_id=trace_id)
        # BudgetExceededError бросается из вызова log_*(), который переводит
        # эту трассу за $1.00 или 20 вызовов, смотря что наступит раньше —
        # вызов, который это спровоцировал, всё равно логируется (он уже
        # произошёл и уже стоил денег к моменту выполнения log_call());
        # исключение — это сигнал остановить цикл, а не способ отменить
        # тот последний вызов.
```

`guard()` останавливает зациклившегося агента в момент превышения лимита
— но он невидим между процессами (два экземпляра `CostTracker`,
разделяющие `trace_id`, даже на одной машине, не видят друг друга) и
забывает всё в момент выхода из блока `with`. Это не дневной/месячный
бюджет.

В счёт идут только вызовы, залогированные с `trace_id`, совпадающим с
активным блоком `guard()` — передавайте `trace_id`, который возвращает
`guard()` (или свой собственный, через `guard(trace_id=...)`), в каждый
вызов `log_call()`/адаптера внутри блока. Два блока `guard()` с разными
`trace_id`, даже вложенные или конкурентные, ведут полностью независимые
суммы. Должен быть задан хотя бы один из `max_usd_per_trace`/
`max_calls_per_trace` — вызов `guard()` без обоих молча ничего бы не
принуждал.

## Используя оба вместе

Типичная настройка: `guard()` вокруг каждого отдельного прогона агента,
чтобы остановить любой отдельный прогон от траты сверх лимита на прогон,
плюс `budget set`, чтобы также знать, когда *месяц* в целом тяготеет к
превышению бюджета по всем прогонам. Ни один не заменяет другой —
`guard()` не может сказать вам, что вы на темпе превышения месячного
бюджета из-за множества мелких, благополучных прогонов, а
`budget`/`BudgetDetector` не может остановить один зациклившийся прогон
до того, как он завершится.
