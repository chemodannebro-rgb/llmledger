# Бэклог

Этот файл — единый источник истины по вопросу «что дальше» для llmledger.
Он заменяет неформальное разделение на «большую команду» (архитектура/ML/
security-ревью) и «мини-команду» (UI/UX/фронтенд-ревью), использовавшееся
в более ранних сессиях планирования — обе группы объединены ниже в один
состав, пересекающиеся роли слиты, чтобы никто не проверял одно и то же
дважды под разной шапкой.

## Команда

|---|---|---|

Итог: 2 роли убраны как дублирующие (отдельные «Дизайнер» и
«тестировщик фронтенда»), 1 роль добавлена (Директор по продукту) — то же
или лучшее покрытие меньшим числом отдельных ревьюеров на одно изменение.

## Как расставляются приоритеты

- **P0** — берём в работу следующим; ценность очевидна, конфликтов
  дизайна нет, внешнего согласования сверх обычного ревью не требуется.
- **P1** — ценно, встаёт в очередь после текущих P0.
- **P2** — приятно иметь / для исследования; берём по возможности.
- **Требует решения** — прежде чем писать код, Директор по продукту
  должен получить явный ответ от владельца проекта, потому что пункт
  конфликтует с принципом, который README уже декларирует как гарантию
  (zero-dependency ядро, отсутствие сетевых вызовов, читаемый плоский лог,
  «портфолио-проект, без поддержки»).

## Раунд 2 бэклога — свежий брейншторм, сверенный с реальным кодом

Пункты 1-12 выше сильно опирались на более ранний аудит кода (пробелы,
которые кто-то уже назвал: PyPI, алертинг, объяснимость, LangChain и
т.д.). Этот второй проход — команда реально перечитывает исходники в
поисках того, что раньше никто не называл — каждый пункт ниже подтверждён
чтением соответствующего файла перед добавлением, а не предположением.

| # | Пункт | Владелец(ы) | Приоритет | Заметки |
|---|---|---|---|---|
| 13 | **`trace_id` фиксируется, но функционально мёртв** | Backend/Core Engineer, Директор по продукту | P1 | Подтверждено в `tracker.py`: каждый адаптер принимает и сохраняет `trace_id`, но `build_report`/`dashboard`/`detect` никогда не читают его обратно — нет способа связать многошаговый вызов (например, retrieval + generation в одном RAG-обороте) в единое число «стоимость этого запроса». Сегодня существует только агрегация `by_label`/`by_model`. Представление `--group-by trace_id` (или поиск `report --trace-id <id>`) сделало бы уже собираемое поле действительно полезным вместо мёртвого груза. |
| 14 | **В baseline-детекции аномалий нет учёта времени суток/дня недели** | ML Engineer, CTO | P2 | Подтверждено в `anomaly/baseline.py`: группировка чисто по `(label, model)`-истории, ничего временного. Легитимная еженедельная batch-задача будет помечаться как аномальная каждую неделю навсегда, так как оценивается против всей истории без сезонной поправки. Настоящее исправление — это изменение самой статистики (не просто недостающий флаг), поэтому CTO должен взвесить это против заявленной философии проекта «простая, объяснимая статистика вместо навороченной модели», прежде чем планировать. |
| 15 | **Нет истории изменений самого `pricing.json`** | Backend/Core Engineer, Technical Writer/Docs | P1 | `pricing.json` несёт только одну дату `last_updated` без записи о том, *что именно* изменилось с предыдущего снапшота. Для инструмента учёта стоимости это реальный пробел в целостности: повторный запуск `report` на том же логе после обновления `pricing.json` может тихо изменить исторические суммы без возможности понять почему. Короткий `PRICING_CHANGELOG.md` (или поле `previous_rate`/`changed` на модель) закрыл бы это дёшево. |
| 16 | **Печатная таблица стилей (`@media print`) для дашборда** | Frontend/UX Engineer | P2 | Раньше не предлагалось. Дашборд уже статический одиночный HTML-файл — печатная таблица стилей (схлопнуть открытые `<details>`, убрать hover/только-тёмную стилизацию) позволила бы передать месячный отчёт стейкхолдеру как чистую печатную страницу/PDF, без новых зависимостей и без JS. |
| 17 | **Нет property-based/фаззинг-тестов для статистики аномалий** | QA/Test Engineer, CTO | P1 | Подтверждено: в `tests/` есть только тесты на конкретных примерах для `_median_mad`/z-score (специально подобранные входные данные). Граничные случаи вроде всех одинаковых значений (MAD=0), групп из одного образца, отрицательных или экстремальных `cost_micros` протестированы по отдельности, но никогда систематически не фаззились. Потребует `hypothesis` как новую **dev-only** зависимость (extra `[dev]`) — не затрагивает поставляемое zero-dependency ядро, но всё равно новая зависимость, которую CTO должен явно одобрить, поскольку проект до сих пор был сознательно минималистичен даже в dev-инструментарии. |
| 18 | **Нет измерения покрытия тестами в CI** | QA/Test Engineer | P2 | Ничто не отслеживает, какую долю кода реально исполняют 123 проходящих теста (например, ветку тёмной темы CSS дашборда или пути ошибок реестра). Добавление `pytest-cov` в `[dev]` и вывод числа в CI (пока без жёсткого гейта, только для наглядности) — низкий риск и могло бы вскрыть непротестированные ветки вроде пункта 17 до того, как они вызовут реальный баг — тот же класс слепого пятна, что позволил мобильному CSS-багу (v0.4.0) уйти мимо существующего набора тестов. |

## Раунд 3 — более глубокий технический аудит (реестр, ML, стриминг, API)

Раунд 2 в основном оставался на поверхности CLI/схемы. Этот проход
углубился построчно в `anomaly/registry.py`, `anomaly/features.py`,
`anomaly/train.py`, `demo_data.py`, `logreader.py` и `pricing.json`. Два
из них (21 и 22) ближе к **рискам корректности, чем к запросам фич**, и
помечены соответственно.

| # | Пункт | Владелец(ы) | Приоритет | Заметки |
|---|---|---|---|---|
| 21 | **Гонка между конкурентными `train` и `detect` на одном `model_dir`** | Backend/Core Engineer, Security Engineer | P0 (риск бага, а не просто фича) | Подтверждено: `latest_version_dir()` (`registry.py`) просто перечисляет наибольший номер версии, без блокировки. Ломающая последовательность: `detect` резолвит «latest = v5» → конкурентный запуск `train` завершается как v6 и обрезает старые версии до `keep_last` → последующий `load_model(v5)` у `detect` теперь попадает на удалённую директорию. Существующий тест (`test_concurrent_saves_from_multiple_threads_get_unique_versions`) доказывает только, что конкурентные *сохранения* не конфликтуют, но не что пара конкурентных train+detect безопасна. Нужен либо паттерн «резолвим номер версии и закрепляем его», либо lock-файл. |
| 22 | **`report`/`detect` материализуют весь лог в список перед обработкой, сводя на нет стриминговый дизайн** | Backend/Core Engineer, CTO | P1 | Подтверждено: `iter_log_records()` (`logreader.py`) — настоящий генератор, но `cmd_report`/`cmd_detect`/`cmd_dashboard` в `cli.py` все делают `records = list(iter_log_records(...))` прежде чем что-либо ещё сделать. На очень большом неротированном логе это загружает весь файл в память как Python-словари — именно та проблема масштаба, которую уже пытается флагировать предупреждение `check_scale()` про «~200k записей», только предупреждение советует ротацию, а не деградацию инструмента изящно. Стоит решить, нужно ли переписать report/detect для накопления агрегатов за один стриминговый проход вместо построения списка сначала (группировка дашборда по дневному журналу усложняет полный отказ от прохода в памяти — CTO должен скоупить это по командам, а не для всего инструмента разом). |
| 23 | **`cached_input_tokens` не является фичей ни в одном из детекторов** | ML Engineer | P1 | Подтверждено: кортеж `FEATURES` в `anomaly/features.py` и оцениваемые поля в `anomaly/baseline.py` — только `("input_tokens", "output_tokens", "cost_micros")` — `cached_input_tokens` логируется, но никогда не оценивается.
